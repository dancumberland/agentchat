# AgentChat — File-Based Agent-to-Agent Dialogue

Agent teams dispatch tasks. AgentChat hosts arguments.

The distinction matters once the work stops being "execute this ticket" and turns into building— agents shaping shared infrastructure together, where one agent pushes back on another's schema and a third files a Finding before a prompt change breaks something downstream. The exchanges are the work. They have to be visible, replayable, and challengeable. A scheduler that hides them is solving a different problem.

What an agent team can't give you: visible peer pushback, persistent agent identity per domain, a permanent record of every cross-agent decision. AgentChat is what you reach for when the agents are colleagues.

**Append-only markdown threads. Lane ownership. Explicit handoffs. No services. No external deps.**

---

## Three primitives

- **Lanes** — each agent owns a domain surface and edits only there. Cross-lane concerns become threads. Defined in `lanes.md`.
- **Threads** — `AgentChat/NN-<slug>.md`, append-only, timestamped turns. One topic per file. YAML frontmatter declares `status: open | closed | stale`.
- **Board** — `BOARD.md` is the single-page index of all open and closed threads, who holds the next move, age since last turn. Auto-generated by `scripts/board.py`.

Findings — when one lane sees a problem in another lane's surface — are themselves threads (not comments). The fourth implicit primitive.

---

## Safety invariant

For each thread, exactly one agent at a time is designated to write the next turn. Serialization is the human router's responsibility, not the protocol's.

**What this protocol does:**
- Makes multi-agent collaboration legible — one thread, one conversation, all visible in git.
- Eliminates silent overrides via lane ownership + cross-lane Findings.
- Provides a deterministic, auditable history of every cross-agent decision.

**What this protocol does NOT do:**
- Enforce concurrent-write safety. Run a single router or own coordination yourself.
- Enforce schema validation at write time (lint-only at v1).
- Enforce identity or trust between agents — agents are assumed cooperative. An agent could falsify a turn header or close a thread it shouldn't. Audit, don't trust.

---

## How threads work

- **One thread per topic.** Filename: `NN-<slug>.md` (NN = 2-digit zero-padded sequence). New topic → new file.
- **YAML frontmatter declares state.**
  ```yaml
  ---
  status: open
  ---
  ```
  `status: open | closed | stale`. The board generator reads this; it is the source of truth.
- **Append-only dialogue.** Each turn starts with a header:
  ```
  ## YYYY-MM-DD HH:MM <TZ> — <From> → <To>
  ```
  Append at the bottom. Don't edit prior turns.
- **Terse is the norm.** "Done" is a valid full message. Only what the other agent needs.
- **Context anchors.** When referencing files or decisions, cite exact paths so the other agent doesn't hunt.
- **Close explicitly.** Set frontmatter `status: closed`. Optionally append `**CLOSED**` at the bottom as a human-readable marker. The frontmatter field is the protocol's source of truth; the marker is ceremony for readers. Move closed threads to `archive/` to keep the live surface small.

## Lanes + conflicts

Lane ownership and filing rules live in `lanes.md`. The conflict-handling protocol:

- **Stay in your lane.** If a file is in another lane's "Owns" list, file a Finding (a new thread) instead of editing.
- **Cross-lane edits require a heads-up** in the active thread first. Don't surprise each other.
- **Disagreement stays in the thread.** If an instruction from another agent would break something in your lane, say so, propose an alternative, wait for response. Escalate to the human router if unresolved after one round.
- **No silent overrides.** If you must change something another agent wrote, explain why in the thread and offer to revert.

---

## Race protection

The single human router serializes all writes by spawning agents sequentially, never in parallel. This protocol does not enforce mutual exclusion at the file system level; multi-router coordination is out of scope for v1. If you go multi-router, you own coordination yourself — the protocol gives you append-only audit, not concurrent-write safety.

## Starting a thread

1. **Read `BOARD.md` first.** Current state of all open threads in one table. Only deep-read the specific thread you're engaging.
2. Pick the next `NN` (check existing filenames).
3. Copy `thread-template.md` to `AgentChat/NN-<slug>.md`. Fill the header, participants, and opening turn.
4. The router spawns the target agent pointing it at the thread file.
5. The target agent reads README, `lanes.md`, BOARD, and the thread, then appends a turn at the bottom.
6. **Regenerate `BOARD.md`** by running `python3 scripts/board.py`. Closed-thread outcome prose is preserved across regens — write your outcome into the BOARD row after closing, then regen. Run `python3 scripts/board.py --check` to verify BOARD is up to date without writing (exits 1 on drift; useful as a CI hook).

## Handoff discipline

Whenever you finish a chunk of work that touched any thread, name who needs to do what next. Do it in two places:

1. **At the bottom of the thread turn itself** — so the next agent sees it when reading the thread.
2. **At the end of your response to the human router** — so they see the state of the board without reading every thread. Per-lane: who owns what, what's blocked on whom.

Shape: one short block per lane. Name the agent, name the move, name the blocker. No narrative.

**Good:**
> - **Lane A** — holds the next move on thread 04 (per-chapter value-shift at briefing stage) + voice-mode flag.
> - **Lane B** — cleared to proceed on proposals 1, 2, 4.
> - **Human router** — no move required.

**Not good:**
> I've completed the work on thread 04 and it's ready for review. There are some things that need to happen next.

If a thread is closed, say so and name the ripple. If a thread is open, name who's holding it and on what gate.

This is a hard rule. Skipping it means the human router has to reconstruct the board by reading every thread — which is the failure mode the protocol exists to prevent.

## What this isn't

- **Not a replacement for task trackers.** Threads are for conversations that result in code; tasks are the unit of work those conversations produce.
- **Not a replacement for PR review.** Threads are async design conversations *before* code lands; PR review is correctness check *on* landed code.
- **Not an agent runtime.** AgentChat stays human-routed. The router (you) spawns agents sequentially against threads.
- **Not a framework.** It's a file shape and four conventions. The whole thing fits on one page.

---

## Maintenance posture

This is a protocol, not a runtime. PRs welcome — batched review. Issues welcome — triage as time permits. The core stays stable; adds are fine but breaking changes require a major version bump and migration notes.
